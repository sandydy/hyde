There are a couple of acronyms and frameworks found so far...

Acronyms

  • AMD, async module definition, is almost a must for javascript module to avoid dependency which introduces javascript initialization precedences
  • MV*, a term I first heard from Ryan Keeter, basically it stands for Model View *, and the * could be ViewModel/ Controler / X (which I dont know yet)

Frameworks & Libraries

There are really a list of libraries available on the internet for each of the areas.

DOM

Many of these libraries exist for a couple of years back to IE5/6 or even earlier. People were looking for single JS library to encapsulate behaviors against different browsers.
  1. JQuery, provides a list of ways to access DOM elements and encapsulate browser specific implementation to programmer, this is obviously a MUST have for next generation web application
  2. DoJo, as far as I know, rival of JQuery, but I've never been using this.
  3. MooTools, framework comes from MIT for a complete OO js. Don't have much time on this, but from time to time I see this lib being used or discussed
  4. Prototype.js, another package to extend DOM. I wouldn't have time to drill into details yet

Model View Controller/ViewModel Frameworks

In recent years especially after HTML 5 was introduced, more logics are shipped to the client. Web client which was originally thin is becoming thicker. These libraries are collections of libraries to address variety of problems. Some of them are MV*  specific, but from what I understand, at least first two of them contains more than just MV*. Simply speaking, all libraries uses HTML 5 data attribute to provide ways to do data-binding.
  1. AngularJS, Google maintained library. Instead of using JQuery like most of the other lib, it contains JQuery light which is a modified version. It also has a list of other packages included, the main idea getting out of it is that AngularJS implements MVC, which is a way Google thinks what a web is. In this article, I am not going to go through details but I would say the way they do binding in the view is cleaner. 
  2. DurandalJS, originally I found only a few information from DurandalJS. Ultimately I found that this is a integration of KnockedOut (MVVM), SammyJS, RequireJS, JQuery, etc. 
  3. BackboneJS

Language Quality Lib

I used to love these libraries as they are telling you something traditionally compiler wont tell, and those problems are hit in runtime. In any case, these libraries are more valuable in javascript as it doesn't have compiler or you will see the problem at the end in browser.Both libraries provide web interface and you can put the js to the web and it will generate a decent report. JSHint is integrated with Node JS. But in general, this is essential to be in javscript continuously integration process when multiple developers are working together. These lib provides cohesive programming style and avoid runtime issues
  1. JSHint, this is fork from JSLint, didn't really drill into details. It has CLI as Node js module. I didn't try, someone said its better but anways, didn't find a good tutorial for usage in windows
  2. JSLint, basically same thing as JSHint. The reason I use this because as my previous post suggested, JSLint can be integrated with Notepad++ (my favourite editor). That why I am using this. 

Modulize Lib

Putting javascript into <script> tag is too difficult to mange when application is becoming larger. RequireJS simply is the one and only I found that majority is using to provide references across files / js modules.
  1. RequireJS, provide patterns to define dependencies per js files. When application is growing big, developer doesn't need to care about which lib is defined or not.

Rounting Lib

I've only tried SammyJS but I remember Ryan Keeter has a couple of others suggested, probably you can follow the reference link at the bottom and try it yourself.
  1. SammyJS, handy lib to define what happens after the hash sign of a page. It simply provides a method for you to provide callback handles when a particular hash value is submitted. It also supports RESTful hash navigation path

Styling, View specific

  1. Bootstrap JS, originated in Twitter and is open source to prepare a set of stylesheets and components for consistent UI styling. It works with DoJo, Jquery, AngularJS. Some suggest that Bootstrap JS was not originally designed for HTML5 CSS3, but at least the round corner is. I will explore a bit more and provide a future post if there are other findings
  2. JQuery UI, one of the very earliest library to provide UI control libraries and cross platform supported, I used to use this for many years. Solid, never disappointed.
  3. MS AJAX, I used to use this lib intensively with ASP.NET with they first introduce client side AJAX control. Components, Controls, etc. they basically provided a way to connect client and server and provide capbility to connect to the web server hosted web service. 

Unit Test

  1. qUnit, Mocha, Jasmine, these are all libraries I've heard from Ryan Keeter, no time to go through them yet. Will find out more details and continue with another blog post.
  2. Selenium, I used to use this library for a couple years to record and replay whatever happen on the browser, this is perfect for stress test, load test, or automated test.

Storage

  1. BreezeJS, AmplifyJS, these are all libraries I've heard from Ryan Keeter, no time to go through them yet. Will find out more details and continue with another blog post.

Backend Server

  1. NodeJS, hottest thing in the industry. Using javascript to provide http server logic via NodeJS, it then uses javascript engine by Google and used in Chrome to provide the capbility. 

Language Supplement Lib

These libraries do not fit in other categories, but I love both of them well. 
  1. Underscore.js, utility library which I found it useful syntactically.
  2. Typescript, I didn't spend too much time with it. But this is a real brilliant idea which provides a high-level syntax to define objects and then compile to javascript. It provides a pretty decent way to avoid ambiguity in javascript as it is always a problem since day one. This is definitely a future of javascript, but just a matter that if the idea could be adopted by majority.
